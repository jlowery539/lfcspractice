Introduction

About the LFCS Certification

The Linux Foundation Certified System Administrator (LFCS) certification validates that a candidate possesses the skills required to perform the day-to-day tasks of a Linux system administrator. It is a vendor-neutral, distribution-flexible certification recognized across the industry. Unlike multiple-choice exams, the LFCS is a performance-based, hands-on exam taken in a live Linux environment — you must demonstrate competence by actually executing commands and configuring real systems.

The exam lasts 2 hours and consists of approximately 20–25 tasks across five weighted domains. Candidates may choose to take the exam on Ubuntu, CentOS/Rocky Linux, or openSUSE. A passing score is 66%.

How to Use This Document

  • Each of the five mock exams is self-contained and designed to be completed within a 2-hour time limit, mirroring the real exam.
  • Attempt all tasks before consulting the solutions. The solutions are placed at the end of each exam to prevent accidental viewing.
  • Each task is labeled with its domain in brackets, e.g., Networking, so you can track your strengths and weaknesses.
  • Tasks are distributed proportionally to the official domain weights: Essential Commands (20%), Users and Groups (10%), Networking (25%), Operations and Deployment (25%), Storage (20%).
  • Use a scratch notepad to record your work — this mirrors the exam environment where you can use a built-in notepad.

Recommended VM Setup

To practice effectively, prepare two virtual machines:

Component Primary VM (node1) Secondary VM (node2)
CPUs 2 vCPUs (minimum) 2 vCPUs (minimum)
RAM 2 GB (minimum) 2 GB (minimum)
Primary Disk 20 GB 20 GB
Extra Disks 3 × 1 GB (for storage labs) 2 × 1 GB (for storage labs)
Network NAT + Host-Only (192.168.1.0/24) NAT + Host-Only (192.168.1.0/24)
OS Ubuntu 22.04+ / Rocky Linux 9+ / AlmaLinux 9+ Same distribution as primary

Hypervisor options: VirtualBox (free, cross-platform), VMware Workstation Player (free for personal use), or KVM/libvirt (native on Linux). Take a snapshot of each VM after initial setup so you can quickly reset between exams.

Tip

Name your VMs node1 and node2. Add entries to /etc/hosts on both machines so they can reach each other by name. Example: 192.168.1.100 node1 and 192.168.1.101 node2.

Essential Commands

Task 1 Essential Commands

Task 4 Essential Commands

An application running on port 8080 is failing to start. Use the systemd journal and system logs to identify the issue. Determine which service is trying to bind to port 8080 and check if another process is already using that port. Document your findings in /root/troubleshoot-8080.txt.

Create the following users and groups:

  • Group developers with GID 3000
  • Group devops with GID 3001
  • User alice (UID 4001) — primary group developers, supplementary group devops, home directory /home/alice, shell /bin/bash

Task 11 Networking

Configure a static route so that traffic destined for the 10.0.0.0/8 network is routed through the gateway 192.168.1.254. Make the route persistent.

Install a container engine (podman or docker), pull the nginx:latest image, and run a container named webserver that maps host port 8080 to container port 80. Ensure the container restarts automatically if it stops. Verify the container is running and accessible.

Exam 1 — Complete Solutions

Solution — Task 1: Git Repository Initialization

Step 1: Create the directory and initialize the repository.

mkdir -p /opt/project
cd /opt/project
git init

This creates a new empty Git repository in /opt/project with the .git directory.

Step 2: Create the file, stage, and commit.

echo "# My Project" > README.md
git add README.md
git commit -m "Initial commit"

git add stages the file for commit. git commit -m creates a snapshot in history.

Step 3: Create the branch and switch to it.

git branch feature-login
git checkout feature-login

Alternatively, in one command:

git checkout -b feature-login

Verification:

git log --oneline
# Expected: a short hash followed by "Initial commit"

git branch
# Expected:
#   main (or master)
# * feature-login

Tip

If git commit fails with "Please tell me who you are", configure your identity first: git config --global user.email "you@example.com" and git config --global user.name "Your Name".

Solution — Task 2: Systemd Service Unit

Step 1: Create the healthcheck script.

cat > /usr/local/bin/healthcheck.sh << 'EOF'
#!/bin/bash
echo "Health check ran at: $(date)" >> /var/log/healthcheck.log
EOF

chmod +x /usr/local/bin/healthcheck.sh

Step 2: Create the systemd service unit file.

cat > /etc/systemd/system/healthcheck.service << 'EOF'
[Unit]
Description=System Health Check Service
After=network.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/healthcheck.sh

[Install]
WantedBy=multi-user.target
EOF

Step 3: Reload systemd, enable, and test.

systemctl daemon-reload
systemctl enable healthcheck.service
systemctl start healthcheck.service

Verification:

systemctl status healthcheck.service
# Should show "Active: inactive (dead)" because it is a oneshot

cat /var/log/healthcheck.log
# Should show a line with the current date

systemctl is-enabled healthcheck.service
# Expected: enabled

Note

A oneshot service runs its ExecStart command and then exits. It is ideal for scripts that perform a single action. The WantedBy=multi-user.target directive means it will run during the normal multi-user boot sequence.

Solution — Task 3: Performance Monitoring

Step 1: Identify top CPU and memory consumers.

# Top 5 by CPU
echo "=== Top 5 Processes by CPU ===" > /root/performance-report.txt
ps aux --sort=-%cpu | head -6 >> /root/performance-report.txt

# Top 5 by Memory
echo "" >> /root/performance-report.txt
echo "=== Top 5 Processes by Memory ===" >> /root/performance-report.txt
ps aux --sort=-%mem | head -6 >> /root/performance-report.txt

Step 2: Add additional performance data.

# System load and uptime
echo "" >> /root/performance-report.txt
echo "=== System Load ===" >> /root/performance-report.txt
uptime >> /root/performance-report.txt

# Memory summary
echo "" >> /root/performance-report.txt
echo "=== Memory Usage ===" >> /root/performance-report.txt
free -h >> /root/performance-report.txt

Step 3: Configure high-CPU logging to syslog.

cat > /usr/local/bin/cpu-monitor.sh << 'EOF'
#!/bin/bash
# Log processes using more than 50% CPU
ps aux | awk '$3 > 50.0 {print}' | while read -r line; do
    logger -t cpu-monitor "High CPU process: $line"
done
EOF

chmod +x /usr/local/bin/cpu-monitor.sh

Schedule this script to run every minute via cron:

echo "* * * * * root /usr/local/bin/cpu-monitor.sh" > /etc/cron.d/cpu-monitor

Verification:

cat /root/performance-report.txt
# Should show formatted process lists

# To test syslog logging, generate CPU load:
# stress --cpu 1 --timeout 120 &
# Wait one minute, then check:
grep "cpu-monitor" /var/log/syslog   # Ubuntu/Debian
grep "cpu-monitor" /var/log/messages  # RHEL-based

Solution — Task 4: Troubleshoot Port 8080

Step 1: Check what is using port 8080.

ss -tlnp | grep 8080
# or
lsof -i :8080

This shows any process currently bound to port 8080.

Step 2: Check systemd journal for relevant errors.

# Look for errors related to port 8080
journalctl -xe | grep 8080

# Check recent errors from all services
journalctl -p err --since "1 hour ago"

# If you know the service name:
journalctl -u your-service-name.service --no-pager -n 50

Step 3: Document findings.

cat > /root/troubleshoot-8080.txt << 'EOF'
=== Port 8080 Troubleshooting Report ===

1. Current process using port 8080:
EOF
ss -tlnp | grep 8080 >> /root/troubleshoot-8080.txt

echo "" >> /root/troubleshoot-8080.txt
echo "2. Journal errors related to port 8080:" >> /root/troubleshoot-8080.txt
journalctl -xe --no-pager | grep 8080 >> /root/troubleshoot-8080.txt 2>&1

echo "" >> /root/troubleshoot-8080.txt
echo "3. All listening ports:" >> /root/troubleshoot-8080.txt
ss -tlnp >> /root/troubleshoot-8080.txt

Resolution: If another process is using port 8080, either stop that process or change the application's configuration to use a different port.

Common Pitfall

The command ss -tlnp flags: -t = TCP, -l = listening, -n = numeric (don't resolve names), -p = show process. You need root/sudo for the -p flag to show process names.

Solution — Task 6: ACLs on Shared Directory

Step 1: Create the directory.

mkdir -p /shared/projects

Step 2: Set ACLs.

# User alice: rwx
setfacl -m u:alice:rwx /shared/projects

# User bob: read-only (r-x for directory traversal)
setfacl -m u:bob:r-x /shared/projects

# Group developers: read and write (rwx for directory)
setfacl -m g:developers:rwx /shared/projects

# Set default ACLs for inheritance
setfacl -d -m u:alice:rwx /shared/projects
setfacl -d -m u:bob:r-- /shared/projects
setfacl -d -m g:developers:rw- /shared/projects

Verification:

getfacl /shared/projects
# Should display all ACL entries including defaults

# Test by creating a file as alice:
su - alice -c "touch /shared/projects/testfile"
getfacl /shared/projects/testfile
# Should inherit default ACLs

Note

The -d flag sets default ACLs, which are inherited by new files and subdirectories created inside the directory. Without default ACLs, only the directory itself gets the permissions, not new content within it.

Solution — Task 7: Static Network Configuration

Method 1: Using nmcli (NetworkManager)

# Delete existing connection profile (if any)
nmcli con delete "Wired connection 1" 2>/dev/null

# Create a new connection with static IPv4
nmcli con add con-name "static-ens33" \
    ifname ens33 \
    type ethernet \
    ipv4.addresses 192.168.1.100/24 \
    ipv4.gateway 192.168.1.1 \
    ipv4.dns "8.8.8.8 8.8.4.4" \
    ipv4.method manual \
    ipv6.addresses fd00::100/64 \
    ipv6.method manual

# Bring the connection up
nmcli con up "static-ens33"

Method 2: Editing Netplan (Ubuntu)

cat > /etc/netplan/01-static.yaml << 'EOF'
network:
  version: 2
  ethernets:
    ens33:
      addresses:
        - 192.168.1.100/24
        - fd00::100/64
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
EOF

netplan apply

Verification:

ip addr show ens33
# Should show 192.168.1.100/24 and fd00::100/64

ip route show default
# Should show "default via 192.168.1.1"

cat /etc/resolv.conf
# Should list 8.8.8.8 and 8.8.4.4

ping -c 2 192.168.1.1
ping6 -c 2 fd00::100

Solution — Task 8: NTP Time Synchronization

Method 1: Using chrony (recommended)

# Install chrony if not present
# RHEL-based:
dnf install -y chrony
# Debian-based:
apt install -y chrony

Edit the chrony configuration:

cat > /etc/chrony.conf << 'EOF'
# Use Google and pool.ntp.org NTP servers
server time.google.com iburst
pool pool.ntp.org iburst

# Record the rate at which the system clock gains/loses time
driftfile /var/lib/chrony/drift

# Allow the system clock to be stepped in the first three updates
makestep 1.0 3

# Enable kernel synchronization of the real-time clock
rtcsync

# Log files location
logdir /var/log/chrony
EOF

systemctl restart chronyd
systemctl enable chronyd

Method 2: Using systemd-timesyncd (simpler, Debian/Ubuntu default)

# Edit the timesyncd config
cat > /etc/systemd/timesyncd.conf << 'EOF'
[Time]
NTP=time.google.com
FallbackNTP=pool.ntp.org
EOF

systemctl restart systemd-timesyncd
timedatectl set-ntp true

Verification:

# For chrony:
chronyc sources -v
chronyc tracking

# For timesyncd:
timedatectl timesync-status
timedatectl status

# Both:
timedatectl
# "System clock synchronized: yes" and "NTP service: active"

Solution — Task 9: SSH Server Hardening

Step 1: Edit the SSH server configuration.

cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak

cat > /etc/ssh/sshd_config.d/custom.conf << 'EOF'
Port 2222
PermitRootLogin no
AllowGroups devops
PasswordAuthentication no
PubkeyAuthentication yes
EOF

Step 2: Restart SSH.

systemctl restart sshd

Step 3: Set up key-based authentication for alice.

# On node1 as alice:
su - alice
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N ""

# Copy the public key to node2:
ssh-copy-id -p 2222 alice@node2
# (If password auth is still enabled on node2 temporarily)

# Alternatively, manually:
# On node2:
mkdir -p /home/alice/.ssh
chmod 700 /home/alice/.ssh
# Copy the contents of node1:/home/alice/.ssh/id_ed25519.pub
# to node2:/home/alice/.ssh/authorized_keys
chmod 600 /home/alice/.ssh/authorized_keys
chown -R alice:developers /home/alice/.ssh

Verification:

# Test SSH on the new port
ssh -p 2222 alice@node2
# Should connect without password prompt

# Verify root login is blocked
ssh -p 2222 root@localhost
# Should be denied

# Verify password auth is disabled
ssh -p 2222 -o PubkeyAuthentication=no alice@localhost
# Should fail with "Permission denied"

Warning

Before disabling password authentication, ensure you have key-based access configured and tested. Otherwise you may lock yourself out. Also update any firewall rules to allow port 2222 instead of 22.

Solution — Task 10: Firewall with nftables

Step 1: Create the nftables ruleset.

cat > /etc/nftables.conf << 'EOF'
#!/usr/sbin/nft -f

flush ruleset

table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;

        # Allow loopback
        iif "lo" accept

        # Allow established/related connections
        ct state established,related accept

        # Allow SSH on port 2222
        tcp dport 2222 accept

        # Allow HTTP and HTTPS
        tcp dport 80 accept
        tcp dport 443 accept

        # Allow ICMP (ping)
        ip protocol icmp accept
        ip6 nexthdr icmpv6 accept

        # Log dropped packets
        log prefix "nftables-drop: " counter drop
    }

    chain forward {
        type filter hook forward priority 0; policy drop;
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}
EOF

Step 2: Load and persist the rules.

# Load the rules
nft -f /etc/nftables.conf

# Enable the nftables service for persistence
systemctl enable nftables
systemctl start nftables

Verification:

# List the ruleset
nft list ruleset

# Test from another machine:
# SSH on 2222 should work
# SSH on 22 should be blocked
# HTTP on 80 should work
# Random port (e.g., 9999) should be blocked

# Check logs for dropped packets:
journalctl -k | grep "nftables-drop"

Solution — Task 13: Process Management

Step 1: Find and investigate the process.

# Find by name
pgrep -a rogue_process
# or
ps aux | grep rogue_process

# Capture the PID
PID=$(pgrep rogue_process)

# Check open files
lsof -p $PID

Step 2: Create the automation script.

cat > /usr/local/bin/kill-rogue.sh << 'SCRIPT'
#!/bin/bash
PROCESS_NAME="rogue_process"

# Find the PID
PID=$(pgrep "$PROCESS_NAME")

if [ -z "$PID" ]; then
    echo "Process '$PROCESS_NAME' not found."
    exit 0
fi

echo "Found $PROCESS_NAME with PID: $PID"

# Show open files
echo "Open files:"
lsof -p "$PID" 2>/dev/null

# Send SIGTERM
echo "Sending SIGTERM to PID $PID..."
kill "$PID"

# Wait up to 10 seconds for graceful shutdown
for i in $(seq 1 10); do
    if ! kill -0 "$PID" 2>/dev/null; then
        echo "Process terminated gracefully."
        exit 0
    fi
    sleep 1
done

# Force kill if still running
echo "Process did not terminate. Sending SIGKILL..."
kill -9 "$PID"
echo "Process force-killed."
SCRIPT

chmod +x /usr/local/bin/kill-rogue.sh

Verification:

# Test with a dummy process:
sleep 3600 &
# Rename conceptually or adjust the script to target "sleep"
# then run the script and confirm the process is killed.

Solution — Task 14: Scheduled Jobs

Step 1: Create the cron job for alice.

# Create a simple backup script first
cat > /home/alice/backup.sh << 'EOF'
#!/bin/bash
tar czf /home/alice/backup-$(date +%Y%m%d).tar.gz /home/alice/Documents 2>/dev/null
EOF
chmod +x /home/alice/backup.sh
chown alice:developers /home/alice/backup.sh

# Add the cron job
crontab -u alice -e
# Add this line:
# 30 2 * * * /home/alice/backup.sh

# Or non-interactively:
(crontab -u alice -l 2>/dev/null; echo "30 2 * * * /home/alice/backup.sh") | crontab -u alice -

Step 2: Create the systemd timer.

# Create the cleanup script
cat > /usr/local/bin/cleanup.sh << 'EOF'

        timeout 10s
    }

        type ipv4_addr

        timeout 10s
    }

        type ipv4_addr
        flags timeout
        timeout 30s

        ct state established,related accept
        ct state invalid drop

        # SYN flood protection
        tcp flags syn limit rate 25/second burst 50 accept
        tcp flags syn drop

        tcp dport 7000 add @knock_stage1 { ip saddr } drop

        # Rate-limited SSH fallback (port 2222)

        # HTTP/HTTPS
        tcp dport { 80, 443 } accept

        # ICMP

        # Log and drop
        log prefix "nft-drop: " counter drop
    }

Solution — Task 10: Complete Firewall/Router with DMZ

# Network layout:
# ens33 = external (203.0.113.1)
# ens34 = internal (192.168.1.1/24)
# ens35 = DMZ (172.16.0.1/24)

sysctl -w net.ipv4.ip_forward=1

cat > /etc/nftables.conf << 'EOF'
#!/usr/sbin/nft -f
flush ruleset

table ip nat {
    chain prerouting {
        type nat hook prerouting priority -100; policy accept;
        # DNAT: external:443 -> DMZ web server
        iifname "ens33" tcp dport 443 dnat to 172.16.0.10:443
        # DNAT: external:25 -> DMZ mail server
        iifname "ens33" tcp dport 25 dnat to 172.16.0.11:25
    }

    chain postrouting {
        type nat hook postrouting priority 100; policy accept;
        # SNAT for internal network
        oifname "ens33" ip saddr 192.168.1.0/24 masquerade
        # SNAT for DMZ
        oifname "ens33" ip saddr 172.16.0.0/24 masquerade
    }
}

table ip filter {
    chain log_and_drop {
        log prefix "fw-drop: " counter drop
    }

    chain input {
        type filter hook input priority 0; policy drop;
        iif "lo" accept
        ct state established,related accept
        # SSH from internal only
        iifname "ens34" tcp dport 22 accept
        ip protocol icmp accept
        jump log_and_drop
    }

    chain forward {
        type filter hook forward priority 0; policy drop;
        ct state established,related accept

        # Internal -> External (full access)
        iifname "ens34" oifname "ens33" accept

        # Internal -> DMZ (full access)
        iifname "ens34" oifname "ens35" accept

        # External -> DMZ (only allowed services)
        iifname "ens33" oifname "ens35" ip daddr 172.16.0.10 tcp dport 443 accept
        iifname "ens33" oifname "ens35" ip daddr 172.16.0.11 tcp dport 25 accept

        # DMZ -> External (limited: DNS, HTTP, HTTPS)
        iifname "ens35" oifname "ens33" tcp dport { 80, 443 } accept
        iifname "ens35" oifname "ens33" udp dport 53 accept

        # DMZ -> Internal: BLOCKED (security isolation)
        iifname "ens35" oifname "ens34" jump log_and_drop

        jump log_and_drop
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}
EOF

systemctl enable nftables

Appendix C: Exam Day Tips

Time Management

  • First pass (60 minutes): Work through all tasks sequentially. Complete the ones you are confident about. Skip any task that seems overly complex or unfamiliar — mark it for later.
  • Second pass (40 minutes): Return to skipped tasks. Attempt partial solutions where possible — partial credit is often awarded.
  • Final review (20 minutes): Verify your work. Run verification commands for each completed task. Check that persistent configurations survive a simulated reboot scenario (review /etc/fstab, systemd enabled services, sysctl.d files).
  • Spend roughly 5–6 minutes per task on average. If a task takes more than 10 minutes, move on.

Common Mistakes to Avoid

Mistake How to Avoid It
Forgetting to make changes persistent Always add entries to /etc/fstab, sysctl.d, or enable systemd units. Verify with systemctl is-enabled.
Not running systemctl daemon-reload Always run it after creating or modifying unit files.
Wrong file permissions Swap files need chmod 600. SSH keys need chmod 600 (private) and chmod 644 (public). The ~/.ssh directory needs chmod 700.
Forgetting restorecon after SELinux context changes Always run restorecon -Rv /path after semanage fcontext.
Typos in configuration files Use validation commands: nginx -t, sshd -t, nft -c -f file, mount -a.
Locking yourself out via SSH Always test SSH changes from a second terminal before closing your current session.
Not checking which distribution you are on Run cat /etc/os-release first. Commands differ between RHEL and Debian families.

How Scoring Works

  • The LFCS exam is scored automatically by checking the state of the system after the exam period ends.
  • The passing score is 66%.
  • Tasks are weighted according to their domain weights: Networking and Operations carry the most weight at 25% each.
  • Partial credit may be awarded — a partially completed task is better than an unattempted one.
  • The system checks end state, not process. It does not matter how you achieve the result (GUI, CLI, or file editing) — only that the final configuration is correct.

What to Do If You Get Stuck

  • Use the man pages. The exam environment has full man and info pages. Use man -k keyword to search. For example: man -k nftables, man 5 fstab.
  • Use built-in help. Most commands support --help. For example: nmcli con add --help, podman run --help.
  • Use tab completion. Bash completion is usually available for systemctl, nmcli, virsh, and other tools.
  • Check examples in config files. Default configuration files often contain commented-out examples. Read /etc/ssh/sshd_config, /etc/chrony.conf, /etc/exports, etc.
  • Break the problem down. If a complex task feels overwhelming, solve it in small pieces. Get the basic version working first, then add refinements.
  • Move on and return. Do not spend more than 10 minutes on a single task. Mark it mentally, solve easier tasks to build confidence, and come back with fresh eyes.

Important

The real exam does not provide internet access. You cannot search the web during the exam. Practice using man pages and --help flags extensively before exam day so you are comfortable finding information offline.
