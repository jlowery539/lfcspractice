Introduction

Purpose of This Lab Manual

This manual provides 100 structured, hands-on lab exercises covering the Networking domain of the Linux Foundation Certified System Administrator (LFCS) exam. Each lab is designed to be completed inside Linux virtual machines, giving you safe, repeatable environments to practice real-world networking tasks without risking production systems.

The labs progress from foundational IPv4/IPv6 configuration through advanced topics such as NAT, load balancing, and reverse proxies. By completing all 100 labs, you will develop the practical networking skills required to pass the LFCS exam and to administer Linux networks in professional environments.

Prerequisites

  • Hypervisor: A working virtualization platform such as VirtualBox, KVM/libvirt, or VMware Workstation.
  • Virtual Machines: At least two VMs — one running CentOS/RHEL 9 and one running Ubuntu 22.04 or 24.04. Many labs require a second (or third) VM as a peer, router, or server. Create snapshots before each lab for easy rollback.
  • Networking: VMs should be configured with at least one NAT adapter (for internet access) and one Host-Only or Internal Network adapter (for inter-VM communication).
  • Basic Linux Knowledge: Comfort with the Linux command line, file editing with vi/nano, package installation (dnf/apt), and systemctl for service management.
  • Root or Sudo Access: All labs assume you have sudo privileges or are logged in as root.

How to Use the Labs

  1. Read the Objective to understand what the lab teaches.
  2. Check Prerequisites/Environment to ensure your VMs are properly configured.
  3. Follow the Step-by-Step Procedure carefully, typing each command exactly as shown.
  4. Verify the Expected Outcome at the end of each lab to confirm success.
  5. Experiment: After completing a lab, modify parameters, break configurations deliberately, and troubleshoot — this deepens your understanding.

Tip

Take VM snapshots before each lab. If something goes wrong, you can revert instantly and try again. Label snapshots clearly (e.g., "Before Lab 12").

Important

Commands prefixed with sudo require elevated privileges. On RHEL/CentOS, many labs use dnf for package management. On Ubuntu, use apt. The labs note distribution-specific differences where applicable.

Section 1: IPv4 Configuration (Labs 1–8)

Covers static and dynamic IPv4 addressing, subnet configuration, multiple IPs, and persistent network settings using nmcli, ip, and Netplan.

Lab 1: Displaying Current IPv4 Configuration

Objective: View all current IPv4 addresses, subnet masks, and default gateways on all network interfaces.

Environment: Any Linux VM (CentOS/RHEL 9 or Ubuntu 22.04/24.04) with at least one active network interface.

Procedure

  1. Display all IP addresses and interface details: 
    ip addr show
    Examine the output. Each interface section begins with a number and interface name (e.g., 2: eth0). Look for inet lines showing IPv4 addresses in CIDR notation (e.g., 192.168.1.10/24).
  2. Display the routing table to find default gateways and connected routes: 
    ip route show
    The line beginning with default via indicates your default gateway.
  3. Display detailed device information via NetworkManager: 
    nmcli device show
    This shows IP addresses, gateways, DNS servers, and connection state for each interface managed by NetworkManager.
  4. View configured DNS servers: 
    cat /etc/resolv.conf
    Look for nameserver entries listing DNS resolver IP addresses.

Expected Outcome

You see interface names (e.g., eth0, enp0s3), IPv4 addresses with CIDR notation, a default gateway in the routing table, and DNS server entries in /etc/resolv.conf. You can identify which interface connects to which network segment.

Lab 2: Configuring a Static IPv4 Address with nmcli

Objective: Assign a static IPv4 address to a network interface using NetworkManager's command-line tool.

Environment: CentOS/RHEL 9 or Ubuntu with NetworkManager active. Identify your connection name first.

Procedure

  1. List all NetworkManager connections to find your connection name: 
    nmcli con show
  2. Modify the connection to use a static IPv4 address (replace enp0s8 with your connection name): 
    sudo nmcli con mod enp0s8 ipv4.addresses 192.168.1.100/24
sudo nmcli con mod enp0s8 ipv4.gateway 192.168.1.1
sudo nmcli con mod enp0s8 ipv4.dns "8.8.8.8"
sudo nmcli con mod enp0s8 ipv4.method manual
  3. Bring the connection down and back up to apply changes: 
    sudo nmcli con down enp0s8
sudo nmcli con up enp0s8
  4. Verify the new static configuration: 
    ip addr show dev enp0s8
ip route show
cat /etc/resolv.conf

Expected Outcome

The interface displays IP address 192.168.1.100/24. The routing table shows the default gateway as 192.168.1.1. DNS resolution to 8.8.8.8 is confirmed in /etc/resolv.conf.

Lab 3: Configuring a Static IPv4 Address with Netplan (Ubuntu)

Objective: Configure a static IPv4 address using Netplan YAML files on Ubuntu.

Environment: Ubuntu 22.04 or 24.04 VM with Netplan as the network configuration renderer.

Procedure

  1. Open the Netplan configuration file for editing: 
    sudo nano /etc/netplan/01-netcfg.yaml
  2. Replace or add the following content (adjust the interface name as needed): 
    network:
  version: 2
  ethernets:
    enp0s8:
      dhcp4: false
      addresses:
        - 10.0.0.50/24
      routes:
        - to: default
          via: 10.0.0.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
  3. Apply the Netplan configuration: 
    sudo netplan apply
  4. Verify the configuration: 
    ip addr show dev enp0s8
ip route show

Expected Outcome

The Ubuntu system displays the static IP 10.0.0.50/24 on the configured interface, and the routing table shows the default gateway as 10.0.0.1.

Lab 4: Configuring DHCP Client

Objective: Set a network interface to obtain its IPv4 address automatically via DHCP.

Environment: A Linux VM with access to a DHCP server (the hypervisor's NAT adapter typically provides one).

Procedure

  1. On CentOS/RHEL, switch the connection to DHCP and remove any static settings: 
    sudo nmcli con mod enp0s8 ipv4.method auto
sudo nmcli con mod enp0s8 ipv4.addresses ""
sudo nmcli con mod enp0s8 ipv4.gateway ""
sudo nmcli con mod enp0s8 ipv4.dns ""
sudo nmcli con up enp0s8
  2. On Ubuntu with Netplan, edit the configuration file to enable DHCP: 
    network:
  version: 2
  ethernets:
    enp0s8:
      dhcp4: true
    Apply with: 
    sudo netplan apply
  3. Verify the dynamically assigned address: 
    ip addr show dev enp0s8
  4. Check DHCP lease information: 
    ls /var/lib/dhclient/
cat /var/lib/dhcp/dhclient.leases

Expected Outcome

The interface acquires an IP address from the DHCP server. A DHCP lease file is created showing the lease duration, assigned IP, and DHCP server address.

Lab 5: Adding Multiple IPv4 Addresses to a Single Interface

Objective: Configure secondary (alias) IPv4 addresses on a single network interface.

Environment: Any Linux VM. A second VM on the same network segment is useful for testing reachability.

Procedure

  1. Add a persistent secondary IP address using nmcli: 
    sudo nmcli con mod enp0s8 +ipv4.addresses 192.168.1.200/24
sudo nmcli con up enp0s8
  2. Alternatively, add a temporary IP address using the ip command (lost on reboot): 
    sudo ip addr add 192.168.1.201/24 dev enp0s8
  3. Verify that all addresses are assigned to the interface: 
    ip addr show dev enp0s8
  4. From another VM on the same network, test reachability to both addresses: 
    ping -c 3 192.168.1.200
ping -c 3 192.168.1.201

Expected Outcome

The interface displays multiple inet entries. Both secondary addresses respond to ping from another VM on the same subnet.

Lab 6: Configuring Custom Subnet Masks

Objective: Understand and apply non-standard subnet masks (/25, /26, /28) and observe their effect on connectivity.

Environment: Two Linux VMs connected to the same internal/host-only network segment.

Procedure

  1. On VM1, assign an IP with a /26 subnet mask: 
    sudo nmcli con mod enp0s8 ipv4.addresses 172.16.0.10/26
sudo nmcli con mod enp0s8 ipv4.method manual
sudo nmcli con up enp0s8
  2. Calculate the network range for 172.16.0.0/26: the range is 172.16.0.0 – 172.16.0.63 (64 addresses, 62 usable hosts).
  3. On VM2, assign an IP within the same /26 range: 
    sudo nmcli con mod enp0s8 ipv4.addresses 172.16.0.20/26
sudo nmcli con mod enp0s8 ipv4.method manual
sudo nmcli con up enp0s8
  4. Test connectivity between the two VMs: 
    ping -c 3 172.16.0.20    # from VM1
ping -c 3 172.16.0.10    # from VM2
  5. Now change VM2 to a /25 mask and observe: 
    sudo nmcli con mod enp0s8 ipv4.addresses 172.16.0.20/25
sudo nmcli con up enp0s8
    Try pinging VM1 again. Connectivity may appear to work in one direction due to mismatched subnet definitions, but routing behavior will be inconsistent.

Expected Outcome

VMs on the same /26 subnet ping each other successfully. When one VM is changed to /25, the subnet mismatch causes unpredictable or failed connectivity, demonstrating the importance of consistent subnet configuration.

Lab 7: Disabling and Enabling Network Interfaces

Objective: Administratively bring network interfaces up and down using both ip and nmcli commands.

Environment: Any Linux VM. Use a secondary interface (not your SSH session's interface) to avoid locking yourself out.

Procedure

  1. Bring an interface down using the ip command: 
    sudo ip link set enp0s8 down
  2. Verify the interface state: 
    ip link show enp0s8
    The output should not contain the UP flag.
  3. Bring the interface back up: 
    sudo ip link set enp0s8 up
  4. Alternatively, use NetworkManager to disconnect and reconnect: 
    sudo nmcli device disconnect enp0s8
sudo nmcli device connect enp0s8
  5. Observe the difference: ip link set down only brings the link down at Layer 2, while nmcli device disconnect also deactivates the NetworkManager connection profile. After ip link set up, you may need to re-apply the connection with nmcli con up to restore IP configuration.

Expected Outcome

The interface state toggles between UP and DOWN. Network connectivity is interrupted when the interface is down and restored when it is brought back up. You understand the behavioral differences between ip and nmcli approaches.

Lab 8: Configuring Persistent Static Routes for IPv4

Objective: Add a static route that persists across reboots.

Environment: A Linux VM with a gateway available to forward traffic to the target network.

Procedure

  1. Add a persistent static route using nmcli: 
    sudo nmcli con mod enp0s8 +ipv4.routes "10.10.0.0/16 192.168.1.1"
sudo nmcli con up enp0s8
  2. Verify the route is in the routing table: 
    ip route show
    You should see a line like: 10.10.0.0/16 via 192.168.1.1 dev enp0s8.
  3. For comparison, add a temporary route using the ip command (this will not survive reboot): 
    sudo ip route add 10.20.0.0/16 via 192.168.1.1
  4. Reboot the VM and check routes again: 
    sudo reboot
    After reboot: 
    ip route show

Expected Outcome

The nmcli-configured route (10.10.0.0/16) persists after reboot. The ip route add temporary route (10.20.0.0/16) is gone after reboot.

Lab 9: Viewing IPv6 Addresses and Understanding Address Types

Objective: Display and identify link-local, global, and loopback IPv6 addresses on your system.

Environment: Any Linux VM with IPv6 enabled (default on most distributions).

Procedure

  1. Display all IPv6 addresses on all interfaces: 
    ip -6 addr show
  2. Identify the address types in the output:
    • fe80:: prefix — Link-local address (auto-generated, used for local segment communication).
    • ::1 — Loopback address (IPv6 equivalent of 127.0.0.1).
    • 2001:, 2600:, etc. — Global unicast addresses (routable on the internet).
  3. View the IPv6 routing table: 
    ip -6 route show
  4. Examine the address scope field in the output. Scope values include link (link-local), global (globally routable), and host (loopback).

Expected Outcome

You can identify link-local (fe80::), loopback (::1), and any global IPv6 addresses on your interfaces. You understand the meaning of each address scope.

The interface displays the assigned global IPv6 address 2001:db8::10/64 with scope global. The link-local address remains present alongside the static global address.

Lab 11: Configuring IPv6 with SLAAC

Objective: Enable Stateless Address Autoconfiguration (SLAAC) to automatically obtain an IPv6 address.

Environment: A Linux VM on a network with an IPv6 router advertisement daemon (radvd) or another VM configured as an IPv6 router.

Procedure

  1. Set the IPv6 method to automatic (SLAAC): 
    sudo nmcli con mod enp0s8 ipv6.method auto
sudo nmcli con up enp0s8
  2. If no IPv6 router is available, set up radvd on another VM. Install and configure: 
    sudo dnf install radvd    # RHEL/CentOS
sudo apt install radvd    # Ubuntu
    Edit /etc/radvd.conf: 
    interface enp0s8
{
    AdvSendAdvert on;
    prefix 2001:db8::/64
    {
        AdvOnLink on;
        AdvAutonomous on;
    };
};
    Start radvd: 
    sudo systemctl enable --now radvd
  3. On the client VM, verify the auto-configured address: 
    ip -6 addr show dev enp0s8

Expected Outcome

The interface receives an auto-configured global IPv6 address from the 2001:db8::/64 prefix based on router advertisements. Both the SLAAC-assigned address and the link-local address are visible.

Lab 12: Setting Up a Dual-Stack (IPv4 + IPv6) Interface

Objective: Configure both IPv4 and IPv6 addresses on the same interface simultaneously.

Environment: A Linux VM with a peer VM configured with both IPv4 and IPv6 on the same segment.

Procedure

  1. Configure both protocols via nmcli: 
    sudo nmcli con mod enp0s8 ipv4.addresses 192.168.1.50/24
sudo nmcli con mod enp0s8 ipv4.method manual

The interface shows both an IPv4 address (192.168.1.50/24) and an IPv6 address (2001:db8::50/64). Both ping (IPv4) and ping -6 (IPv6) succeed to their respective peers.

Lab 13: Disabling IPv6 on a Specific Interface

Objective: Disable IPv6 on a single interface using sysctl without affecting other interfaces.

Environment: Any Linux VM with multiple interfaces (so you can observe that disabling IPv6 on one leaves others unaffected).

Procedure

  1. Disable IPv6 on a specific interface at runtime: 
    sudo sysctl -w net.ipv6.conf.enp0s8.disable_ipv6=1
  2. Verify that IPv6 is disabled on that interface: 
    ip -6 addr show dev enp0s8
  3. Confirm other interfaces still have IPv6: 
    ip -6 addr show dev enp0s3
  4. Make the change persistent by creating a sysctl configuration file: 
    echo "net.ipv6.conf.enp0s8.disable_ipv6 = 1" | sudo tee /etc/sysctl.d/99-disable-ipv6.conf
sudo sysctl --system
  5. To re-enable IPv6 later: 
    sudo sysctl -w net.ipv6.conf.enp0s8.disable_ipv6=0

Expected Outcome

No IPv6 address appears on the target interface (enp0s8). Other interfaces retain their IPv6 addresses. The setting persists across reboots when saved to /etc/sysctl.d/.

Lab 14: Testing IPv6 Connectivity with ping6 and traceroute6

Objective: Verify IPv6 reachability and trace the network path using IPv6-specific diagnostic tools.

Successful ping replies over IPv6 are received. The traceroute output shows each hop along the IPv6 path. Link-local pings work when the correct scope ID (%interface) is specified.

Lab 18: Configuring DNS Client Settings with /etc/resolv.conf

Objective: Manually configure DNS servers and search domains for the system.

Environment: Any Linux VM with NetworkManager active.

Procedure

  1. Understand that NetworkManager manages /etc/resolv.conf. Direct edits are overwritten. Use nmcli instead: 
    sudo nmcli con mod enp0s8 ipv4.dns "8.8.8.8 8.8.4.4"

sudo nmcli con up enp0s8
  2. Verify the resulting resolv.conf: 
    cat /etc/resolv.conf
    You should see nameserver 8.8.8.8, nameserver 8.8.4.4, and search lab.local example.com.
  3. Test DNS resolution: 
    nslookup google.com
dig google.com

Expected Outcome

/etc/resolv.conf shows the configured nameservers and search domains. DNS queries via nslookup and dig return valid results from the configured DNS servers.

Lab 19: Understanding and Configuring NSS with /etc/nsswitch.conf

Objective: Control the order in which hostname resolution sources are consulted.

Environment: Any Linux VM with entries in both /etc/hosts and DNS for the same hostname (to see the effect of ordering).

Procedure

  1. View the current NSS configuration: 
    cat /etc/nsswitch.conf | grep hosts
    The default line is typically: hosts: files dns myhostname.
  2. Add a test entry in /etc/hosts that maps google.com to 127.0.0.1 (for demonstration purposes only): 
    echo "127.0.0.1 google.com" | sudo tee -a /etc/hosts
  3. Test resolution — with files first, the hosts file entry takes priority: 
    getent hosts google.com
  4. Edit /etc/nsswitch.conf and change the hosts line to dns files myhostname: 
    sudo nano /etc/nsswitch.conf
  5. Test again: 
    getent hosts google.com
    Now DNS is queried first, returning the real IP address.
  6. Reset the hosts line to its default: files dns myhostname, and remove the test entry from /etc/hosts.

Expected Outcome

When files is listed first, /etc/hosts entries take priority over DNS. When dns is first, DNS responses override local entries. You understand the lookup sequence defined by NSS.

Lab 20: Using dig, nslookup, and host for DNS Troubleshooting

Objective: Query specific DNS record types using standard command-line tools.

Environment: Any Linux VM with internet access and DNS tools installed.

Procedure

  1. Query A records (IPv4 address): 
    dig example.com A
  2. Query MX records (mail servers): 
    dig example.com MX
  3. Query NS records (name servers): 
    dig example.com NS
  4. Perform a reverse DNS lookup: 
    dig -x 8.8.8.8
  5. Use nslookup for a simpler query: 
    nslookup example.com
  6. Use host for a concise result: 
    host example.com
  7. Compare the output formats. dig provides the most detail (query time, flags, authority section), while host is the most concise.

Expected Outcome

You retrieve A, MX, NS, and PTR records successfully. You understand the query/response structure of DNS and can interpret the ANSWER, AUTHORITY, and ADDITIONAL sections in dig output.

Environment: A VM with at least two NICs (enp0s8, enp0s9) and VMs connected to each.

Procedure

  1. Create bridge and add both interfaces: 
    sudo ip link add name br0 type bridge
sudo ip link set enp0s8 master br0
sudo ip link set enp0s9 master br0
sudo ip link set br0 up
sudo ip link set enp0s8 up
sudo ip link set enp0s9 up
  2. Remove any IP addresses from the member interfaces (IP goes on the bridge): 
    sudo ip addr flush dev enp0s8
sudo ip addr flush dev enp0s9
sudo ip addr add 192.168.1.1/24 dev br0
  3. From VMs connected to each interface, ping each other to verify Layer 2 bridging.
  4. Check the MAC address table: 
    sudo brctl showmacs br0

Expected Outcome

VMs on both physical interfaces are on the same Layer 2 segment and communicate directly. The MAC address table shows learned addresses on each port.

Lab 84: Configuring STP on a Linux Bridge

Objective: Enable and configure Spanning Tree Protocol to prevent Layer 2 loops.

Environment: A Linux VM with a bridge configured (from Lab 82 or 83).

Procedure

  1. Enable STP on the bridge: 
    sudo brctl stp br0 on
    Or using ip: 
    sudo ip link set br0 type bridge stp_state 1
  2. Set the bridge priority (lower is preferred as root bridge): 
    sudo brctl setbridgeprio br0 8192
  3. Set path cost for a port: 
    sudo brctl setpathcost enp0s9 10
  4. View STP state: 
    sudo brctl showstp br0

Expected Outcome

STP is active on the bridge. Port states (forwarding, blocking, listening, learning) are visible. The bridge priority and path costs influence root bridge election and port roles.

Lab 85: Creating a Bridge for KVM/QEMU Virtual Machines

Objective: Set up a bridge that KVM VMs attach to for direct access to the physical network.

Environment: A KVM/libvirt host with at least one physical NIC.

Procedure

  1. Create the bridge with nmcli and add the physical interface: 
    sudo nmcli con add type bridge ifname br0 con-name br0
sudo nmcli con add type bridge-slave ifname enp0s3 master br0
sudo nmcli con up br0
  2. When creating KVM VMs, specify the bridge network: 
    virt-install --name testvm --ram 1024 --disk size=10 --network bridge=br0 --cdrom /path/to/iso
  3. Verify the VM gets an IP on the physical network: 
    sudo virsh domiflist testvm
bridge link show

Expected Outcome

KVM virtual machines have direct Layer 2 access to the physical network through the bridge. The VMs obtain IP addresses from the physical network's DHCP server (or use static IPs on the same subnet).

Lab 86: Persistent Bridge Configuration with Netplan (Ubuntu)

Objective: Configure a bridge declaratively using Netplan on Ubuntu.

The bridge br0 is created at boot via Netplan with both enp0s8 and enp0s9 as member ports and the correct IP address assigned.

Section 12: Bonding Interfaces (Labs 87–91)

Covers network interface bonding (teaming) for redundancy and increased throughput.

Lab 87: Creating a Bond Interface with nmcli (Mode 1 — Active-Backup)

Objective: Bond two NICs for failover redundancy using active-backup mode.

Environment: A Linux VM with at least two NICs.

Procedure

  1. Create the bond interface: 
    sudo nmcli con add type bond ifname bond0 con-name bond0 \
  bond.options "mode=active-backup,miimon=100" \
  ipv4.addresses 192.168.1.10/24 ipv4.method manual
  2. Add slave interfaces: 
    sudo nmcli con add type bond-slave ifname enp0s8 master bond0
sudo nmcli con add type bond-slave ifname enp0s9 master bond0
  3. Bring up the bond: 
    sudo nmcli con up bond0
  4. Verify the bond status: 
    cat /proc/net/bonding/bond0

Expected Outcome

The bond bond0 is active with one NIC as the active slave and the other as backup. The bond IP (192.168.1.10/24) is reachable. The /proc/net/bonding/bond0 file shows both slaves and their states.

Lab 88: Configuring Bond Mode 0 (Balance Round-Robin)

Objective: Set up round-robin bonding for increased aggregate throughput.

Environment: A Linux VM with at least two NICs.

Procedure

  1. Create the bond with round-robin mode: 
    sudo nmcli con add type bond ifname bond0 con-name bond0 \
  bond.options "mode=balance-rr,miimon=100" \
  ipv4.addresses 192.168.1.10/24 ipv4.method manual
  2. Add slave interfaces and bring up.
  3. Verify: 
    cat /proc/net/bonding/bond0
  4. Test throughput with iperf3 and observe traffic distribution: 
    ip -s link show bond0

Expected Outcome

Traffic is distributed across both NICs in round-robin fashion. Aggregate throughput may increase compared to a single NIC (dependent on switch configuration and traffic patterns).

Lab 89: Configuring Bond Mode 4 (802.3ad / LACP)

Objective: Set up LACP bonding for standards-based link aggregation.

Environment: A Linux VM with at least two NICs. Full LACP requires switch support.

Procedure

  1. Create the bond with LACP mode: 
    sudo nmcli con add type bond ifname bond0 con-name bond0 \
  bond.options "mode=802.3ad,miimon=100,lacp_rate=fast" \
  ipv4.addresses 192.168.1.10/24 ipv4.method manual
  2. Add slave interfaces and bring up.
  3. Verify LACP negotiation: 
    cat /proc/net/bonding/bond0
    Look for "LACP rate:" and "Partner" information.

Expected Outcome

LACP negotiation succeeds if the switch supports 802.3ad. The bond status shows LACP partner information. In a VM-only lab without switch support, the bond may show that LACP negotiation is pending — this is normal and educational.

Lab 90: Testing Bond Failover

Objective: Verify that active-backup bonding maintains connectivity when a slave fails.

Environment: A VM with an active-backup bond from Lab 87 and a peer VM for continuous ping.

Procedure

  1. Start a continuous ping from the peer VM: 
    ping 192.168.1.10
  2. Identify the currently active slave: 
    cat /proc/net/bonding/bond0 | grep "Currently Active"
  3. Simulate a link failure by bringing down the active slave: 
    sudo ip link set enp0s8 down
  4. Observe the ping — after a brief interruption (based on miimon), it should resume.
  5. Check which slave is now active: 
    cat /proc/net/bonding/bond0 | grep "Currently Active"
  6. Bring the original slave back up: 
    sudo ip link set enp0s8 up

Expected Outcome

Network connectivity continues during failover with only a brief interruption (≤ miimon interval). The active slave switches automatically. Bringing the failed interface back up restores it as a backup slave.

Lab 91: Bonding with VLAN Tagging

Objective: Configure a VLAN on top of a bonded interface for segmented traffic.

Procedure

  2. Bring the VLAN interface up:

  4. Test connectivity to another VM on VLAN 100:

VLAN 100 is active on top of bond0. The VLAN interface has its own IP address (10.100.0.10/24). Traffic is tagged with VLAN ID 100 on the wire. The bond provides redundancy while the VLAN provides network segmentation.

Section 13: Reverse Proxies (Labs 92–96)

Covers configuring Nginx and Apache as reverse proxies with TLS termination and basic access control.

Lab 92: Installing and Configuring Nginx as a Basic Reverse Proxy

Objective: Set up Nginx to proxy HTTP requests to a backend server.

Environment: A frontend VM with Nginx and a backend VM (or local process) running a web server on port 8080.

Procedure

  1. Install Nginx: 
    sudo dnf install nginx    # RHEL/CentOS
sudo apt install nginx    # Ubuntu
  2. Start a backend web server (for testing): 
    python3 -m http.server 8080 &
  3. Create the reverse proxy configuration: 
    sudo nano /etc/nginx/conf.d/reverse-proxy.conf
    Add: 
    server {
    listen 80;
    server_name _;

    location / {
        proxy_pass http://192.168.1.200:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
  4. Test and restart Nginx: 
    sudo nginx -t
sudo systemctl restart nginx
  5. Test the proxy: 
    curl http://localhost

Expected Outcome

Requests to Nginx on port 80 are forwarded to the backend on port 8080. The response content comes from the backend server. The backend receives the real client IP in the X-Real-IP header.

Lab 93: Configuring Nginx Reverse Proxy with Multiple Backends (Path-Based)

Objective: Route different URL paths to different backend servers.

Environment: An Nginx VM and two backend VMs (or two local processes on different ports).

Procedure

  1. Set up two backend servers on different VMs or ports: 
    # Backend 1
cd /tmp/app1 && python3 -m http.server 8081 &
# Backend 2
cd /tmp/app2 && python3 -m http.server 8082 &
  2. Configure path-based routing in Nginx: 
    server {
    listen 80;

    location /app1/ {
        proxy_pass http://192.168.1.201:8081/;
    }

    location /app2/ {
        proxy_pass http://192.168.1.202:8082/;
    }
}
  3. Restart and test: 
    sudo nginx -t && sudo systemctl restart nginx
curl http://localhost/app1/
curl http://localhost/app2/

Expected Outcome

Requests to /app1/ are proxied to backend 1, and requests to /app2/ are proxied to backend 2. Each path serves content from its respective backend.

Lab 94: Setting Up TLS Termination with Nginx

Objective: Configure HTTPS on the reverse proxy using a self-signed certificate for TLS termination.

Environment: An Nginx VM with OpenSSL installed.

Procedure

  1. Create the SSL directory and generate a self-signed certificate: 
    sudo mkdir -p /etc/nginx/ssl
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/nginx/ssl/server.key \
  -out /etc/nginx/ssl/server.crt \
  -subj "/CN=lab.example.com"
  2. Configure Nginx for HTTPS with HTTP-to-HTTPS redirect: 
    server {
    listen 80;
    server_name lab.example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name lab.example.com;

    ssl_certificate /etc/nginx/ssl/server.crt;
    ssl_certificate_key /etc/nginx/ssl/server.key;

    location / {
        proxy_pass http://192.168.1.200:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
  3. Restart and test: 
    sudo nginx -t && sudo systemctl restart nginx
curl -k https://localhost

Expected Outcome

HTTPS connections are terminated at Nginx; the backend receives plain HTTP. HTTP requests are automatically redirected to HTTPS. The -k flag in curl bypasses the self-signed certificate warning.

Lab 95: Configuring Apache as a Reverse Proxy

Objective: Use Apache httpd with mod_proxy for reverse proxying.

Environment: A Linux VM with Apache httpd installed.

Procedure

  1. Install Apache: 
    sudo dnf install httpd    # RHEL/CentOS
sudo apt install apache2  # Ubuntu
  2. Enable proxy modules (Ubuntu): 
    sudo a2enmod proxy proxy_http
    On RHEL/CentOS, verify proxy modules are loaded in the httpd configuration.
  3. Create a VirtualHost for the reverse proxy: 
    sudo nano /etc/httpd/conf.d/reverse-proxy.conf
    (Or /etc/apache2/sites-available/reverse-proxy.conf on Ubuntu) 
    <VirtualHost *:80>
    ServerName lab.example.com
    ProxyPreserveHost On
    ProxyPass / http://192.168.1.200:8080/
    ProxyPassReverse / http://192.168.1.200:8080/
</VirtualHost>
  4. Restart Apache and test: 
    sudo systemctl restart httpd    # RHEL/CentOS
sudo systemctl restart apache2  # Ubuntu
curl http://localhost

Expected Outcome

Apache proxies HTTP requests to the backend server. ProxyPassReverse rewrites response headers so redirects work correctly through the proxy.

Lab 96: Adding Access Control to a Reverse Proxy

Objective: Restrict reverse proxy access by IP address and/or basic authentication.

Environment: An Nginx reverse proxy from Lab 92 or 94.

Procedure

  1. Restrict access by IP address in the Nginx location block: 
    location / {
    allow 192.168.1.0/24;
    deny all;
    proxy_pass http://192.168.1.200:8080;
}
  2. Create a password file for basic authentication: 
    sudo dnf install httpd-tools    # RHEL/CentOS
sudo apt install apache2-utils  # Ubuntu
sudo htpasswd -c /etc/nginx/.htpasswd labuser
  3. Add basic auth to the Nginx configuration: 
    location / {
    auth_basic "Restricted Access";
    auth_basic_user_file /etc/nginx/.htpasswd;
    proxy_pass http://192.168.1.200:8080;
}
  4. Restart and test: 
    sudo nginx -t && sudo systemctl restart nginx
curl http://localhost                           # expect 401
curl -u labuser:password http://localhost       # expect 200

Expected Outcome

Requests from unauthorized IPs receive HTTP 403 (Forbidden). Requests without credentials receive HTTP 401 (Unauthorized). Only authenticated users from allowed IPs can access the proxied content.

Lab 97: Configuring Nginx as a Round-Robin HTTP Load Balancer

Objective: Distribute HTTP traffic evenly across multiple backend servers.

Environment: An Nginx VM and three backend VMs (or three local processes), each returning a unique identifier.

Procedure

  1. Set up three backend servers that identify themselves (e.g., each returns its hostname): 
    # On Backend 1 (192.168.1.201):
echo "Backend 1" > /tmp/index.html && cd /tmp && python3 -m http.server 8080 &
# Repeat on Backend 2 (192.168.1.202) and Backend 3 (192.168.1.203)
  2. Configure Nginx upstream and proxy: 
    upstream backend {
    server 192.168.1.201:8080;
    server 192.168.1.202:8080;
    server 192.168.1.203:8080;
}

server {
    listen 80;

    location / {
        proxy_pass http://backend;
    }
}
  3. Restart and test with repeated requests: 
    sudo nginx -t && sudo systemctl restart nginx
for i in $(seq 1 9); do curl -s http://localhost; done

Expected Outcome

Requests are distributed evenly across all three backends in round-robin rotation. The output cycles through "Backend 1", "Backend 2", "Backend 3" repeatedly.

Lab 98: Configuring Nginx Load Balancer with Weighted and Least-Connections Methods

Objective: Use advanced load balancing algorithms to control traffic distribution.

Environment: Nginx VM with multiple backends (from Lab 97).

Procedure

  1. Configure weighted round-robin (backend 1 gets 5x more traffic): 
    upstream backend {
    server 192.168.1.201:8080 weight=5;
    server 192.168.1.202:8080 weight=1;
}
  2. Configure least-connections (sends to the server with fewest active connections): 
    upstream backend {
    least_conn;
    server 192.168.1.201:8080;
    server 192.168.1.202:8080;
}
  3. Configure IP hash (session persistence by client IP): 
    upstream backend {
    ip_hash;
    server 192.168.1.201:8080;
    server 192.168.1.202:8080;
}
  4. Test each method by restarting Nginx and sending multiple requests: 
    sudo nginx -t && sudo systemctl restart nginx
for i in $(seq 1 12); do curl -s http://localhost; done

Expected Outcome

With weighted mode, the higher-weight server receives proportionally more requests. With least-connections, busy servers receive fewer new requests. With IP hash, the same client always reaches the same backend.

Lab 99: Installing and Configuring HAProxy as a Load Balancer

Objective: Set up HAProxy for HTTP load balancing with health checks and a statistics dashboard.

Environment: A Linux VM for HAProxy and two or more backend VMs.

Procedure

  1. Install HAProxy: 
    sudo dnf install haproxy    # RHEL/CentOS
sudo apt install haproxy    # Ubuntu
  2. Edit the HAProxy configuration: 
    sudo nano /etc/haproxy/haproxy.cfg
    Add the following (after the global/defaults sections): 
    frontend http_front
    bind *:80
    default_backend http_back

backend http_back
    balance roundrobin
    server web1 192.168.1.201:8080 check
    server web2 192.168.1.202:8080 check

listen stats
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 10s
  3. Start HAProxy: 
    sudo systemctl enable --now haproxy
  4. Test load balancing: 
    for i in $(seq 1 6); do curl -s http://localhost; done
  5. Access the statistics dashboard in a browser or with curl: 
    curl http://localhost:8404/stats

Expected Outcome

HAProxy distributes traffic across backends in round-robin fashion. Health checks automatically remove failed backends from the rotation. The statistics page on port 8404 shows server statuses, connection counts, and health check results.

Lab 100: HAProxy with Session Persistence and SSL Termination

Objective: Configure cookie-based sticky sessions and HTTPS termination on HAProxy.

Environment: HAProxy VM with backends (from Lab 99) and OpenSSL for certificate generation.

Procedure

  1. Generate a certificate and create a combined PEM file for HAProxy: 
    sudo mkdir -p /etc/haproxy/ssl
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/haproxy/ssl/server.key \
  -out /etc/haproxy/ssl/server.crt \
  -subj "/CN=lab.example.com"
sudo cat /etc/haproxy/ssl/server.crt /etc/haproxy/ssl/server.key > /etc/haproxy/ssl/combined.pem
  2. Update the HAProxy configuration for SSL and sticky sessions: 
    frontend http_front
    bind *:80
    redirect scheme https if !{ ssl_fc }

frontend https_front
    bind *:443 ssl crt /etc/haproxy/ssl/combined.pem
    default_backend http_back

backend http_back
    balance roundrobin
    cookie SERVERID insert indirect nocache
    server web1 192.168.1.201:8080 check cookie web1
    server web2 192.168.1.202:8080 check cookie web2
  3. Restart and test: 
    sudo systemctl restart haproxy
  4. Test HTTP-to-HTTPS redirect: 
    curl -v http://localhost 2>&1 | grep "Location"
  5. Test HTTPS and session persistence: 
    curl -k -c cookies.txt https://localhost
curl -k -b cookies.txt https://localhost
curl -k -b cookies.txt https://localhost
    All three requests with the cookie should go to the same backend.

Expected Outcome

HTTP requests are automatically redirected to HTTPS. HTTPS connections are terminated at HAProxy with the self-signed certificate. The SERVERID cookie ensures session persistence — repeated requests with the same cookie always reach the same backend server.
