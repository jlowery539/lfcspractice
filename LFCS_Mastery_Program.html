Chapter 1: Basic Git Operations

1.1 What Is Git and Why Sysadmins Need It

Git is a distributed version control system originally created by Linus Torvalds in 2005 for Linux kernel development. For system administrators, Git is an essential tool that extends far beyond software development. Modern sysadmin workflows rely on Git for:

  • Configuration management: Tracking changes to /etc/ configuration files across servers
  • Infrastructure as Code (IaC): Versioning Ansible playbooks, Terraform configs, Puppet manifests, and shell scripts
  • Change auditing: Maintaining a complete history of who changed what, when, and why
  • Collaboration: Enabling teams to work on shared infrastructure code with branching and merging
  • Disaster recovery: Providing the ability to roll back to a known good configuration state

⚑ Exam Alert

The LFCS exam tests basic Git operations including initializing repositories, staging and committing files, working with branches, and cloning/pushing to remotes. You will not be tested on advanced Git workflows like rebasing interactive mode or bisecting.

1.2 Git Architecture

Git uses a three-stage architecture that every sysadmin must understand:

StageDescriptionKey Commands
Working DirectoryYour actual files on disk. This is where you edit files.git status
Staging Area (Index)A preparation area where you assemble the next commit. Files are "staged" here before being committed.git add
Repository (.git)The permanent storage of all committed snapshots, branches, tags, and metadata.git commit

The basic workflow is: edit files in the working directory → stage changes with git add → save a snapshot with git commit.

1.3 Installing Git

Git is available in the default repositories of all major distributions:

# Debian/Ubuntu
sudo apt update && sudo apt install -y git

# RHEL/Rocky Linux/Fedora
sudo dnf install -y git

# openSUSE
sudo zypper install -y git

Verify the installation:

$ git --version
git version 2.39.2

1.4 Initial Configuration

Before using Git, configure your identity. These settings are stored in ~/.gitconfig:

# Set your name and email (used in every commit)
git config --global user.name "Jane Admin"
git config --global user.email "jane@example.com"

# Set your preferred text editor
git config --global core.editor vim

# Set default branch name
git config --global init.defaultBranch main

# View all configuration
git config --list

# View a specific setting
git config user.name

Git configuration has three levels of scope:

LevelFlagFile LocationScope
System--system/etc/gitconfigAll users on the system
Global--global~/.gitconfigCurrent user, all repositories
Local--local.git/configCurrent repository only

1.5 Creating Repositories

Initializing a New Repository

# Create and initialize a new repository
mkdir /opt/sysconfig
cd /opt/sysconfig
git init

# Output:
# Initialized empty Git repository in /opt/sysconfig/.git/

The git init command creates a hidden .git/ directory containing all repository metadata, objects, and references.

Cloning an Existing Repository

# Clone via HTTPS
git clone https://github.com/example/infra-configs.git

# Clone via SSH
git clone git@github.com:example/infra-configs.git

# Clone into a specific directory
git clone https://github.com/example/infra-configs.git /opt/infra

1.6 The Git Workflow: Edit, Stage, Commit

# Check the status of your working directory
git status

# Stage a specific file
git add filename.conf

# Stage all changed files
git add .
git add -A

# Commit staged changes with a message
git commit -m "Add initial nginx configuration"

# Stage and commit tracked files in one step
git commit -am "Update DNS resolver settings"

💡 Pro Tip

Write commit messages in the imperative mood (e.g., "Add firewall rule" not "Added firewall rule"). This convention matches Git's own generated messages and is the industry standard.

1.7 Branching and Merging

Branches allow you to work on different versions of your configuration simultaneously without affecting the main line:

# List all branches (* marks current branch)
git branch

# Create a new branch
git branch feature/firewall-update

# Switch to a branch
git checkout feature/firewall-update

# Create and switch in one command
git checkout -b feature/firewall-update

# Modern alternative (Git 2.23+)
git switch -c feature/firewall-update

# Merge a branch into the current branch
git checkout main
git merge feature/firewall-update

# Delete a merged branch
git branch -d feature/firewall-update

# Force-delete an unmerged branch
git branch -D feature/firewall-update

Resolving Merge Conflicts

When two branches modify the same lines, Git reports a merge conflict. The conflicting file will contain markers:

<<<<<<< HEAD
PermitRootLogin no
=======
PermitRootLogin yes
>>>>>>> feature/ssh-config

To resolve: edit the file to keep the desired content, remove the conflict markers, then stage and commit:

git add sshd_config
git commit -m "Resolve merge conflict in sshd_config"

1.8 Working with Remotes

# List configured remotes
git remote -v

# Add a remote repository
git remote add origin https://github.com/example/configs.git

# Push local commits to remote
git push origin main

# Push and set upstream tracking
git push -u origin main

# Fetch remote changes (does not merge)
git fetch origin

# Pull remote changes (fetch + merge)
git pull origin main

# Remove a remote
git remote remove origin

1.9 Viewing History

# View commit history
git log

# Compact one-line format
git log --oneline

# Graphical branch view
git log --oneline --graph --all

# Show changes in each commit
git log -p

# Show last 5 commits
git log -5

# View a specific commit
git show abc1234

# View differences
git diff                 # Working dir vs staging area
git diff --staged        # Staging area vs last commit
git diff main..feature   # Between two branches

1.10 Undoing Changes

# Discard changes in working directory (restore from staging area)
git restore filename.conf

# Unstage a file (keep working directory changes)
git restore --staged filename.conf

# Amend the last commit (change message or add files)
git commit --amend -m "Updated commit message"

# Reset to a previous commit (soft: keep changes staged)
git reset --soft HEAD~1

# Reset to a previous commit (mixed: keep changes unstaged)
git reset HEAD~1

# Reset to a previous commit (hard: discard all changes)
git reset --hard HEAD~1

# Create a new commit that reverses a previous commit
git revert abc1234

⚠ Warning

git reset --hard permanently discards uncommitted changes and cannot be undone. Always verify your working directory status with git status before running hard resets.

1.11 The .gitignore File

The .gitignore file tells Git which files and directories to exclude from tracking. Common patterns for sysadmin use:

# Ignore log files
*.log

# Ignore temporary files
*.tmp
*.swp
*~

# Ignore secrets and credentials
*.key
*.pem
secrets/
.env

# Ignore OS files
.DS_Store
Thumbs.db

# Ignore compiled binaries
*.o
*.so

# But DO track a specific file
!important.log

1.12 Git Tags

# Create a lightweight tag
git tag v1.0

# Create an annotated tag (recommended)
git tag -a v1.0 -m "First stable release of configs"

# List all tags
git tag

# Show tag details
git show v1.0

# Push tags to remote
git push origin v1.0
git push origin --tags

# Delete a tag
git tag -d v1.0

1.13 Stashing Changes

# Stash current changes
git stash

# Stash with a descriptive message
git stash push -m "WIP: firewall rule changes"

# List stashes
git stash list

# Apply the most recent stash (keep in stash list)
git stash apply

git stash pop

# Apply a specific stash

# Drop a specific stash
git stash drop stash@{0}

# Clear all stashes
git stash clear

Lab 1.2: Branching and Merging Workflow

Objective: Create a feature branch, make changes, and merge back to main. Then resolve a simulated merge conflict.

Step 1: Create and switch to a feature branch.

cd /opt/sysconfig
git checkout -b feature/ssl-config

Expected output:

Switched to a new branch 'feature/ssl-config'

Step 2: Add an SSL configuration on the feature branch.

cat > ssl.conf <<EOF
# SSL Configuration
ssl_certificate=/etc/ssl/certs/server.crt
ssl_certificate_key=/etc/ssl/private/server.key
ssl_protocols=TLSv1.2 TLSv1.3
ssl_ciphers=HIGH:!aNULL:!MD5
EOF

git add ssl.conf
git commit -m "Add SSL configuration"

Step 3: Switch back to main and make a conflicting change.

git checkout main
# Modify webserver.conf on main
sed -i 's/listen_port=80/listen_port=8080/' webserver.conf
git add webserver.conf
git commit -m "Change listen port to 8080 on main"

Step 4: Now modify the same file on the feature branch to create a conflict.

git checkout feature/ssl-config
sed -i 's/listen_port=80/listen_port=443/' webserver.conf
git add webserver.conf
git commit -m "Change listen port to 443 for SSL"

Step 5: Merge and resolve the conflict.

git checkout main
git merge feature/ssl-config

Expected output (conflict):

Auto-merging webserver.conf
CONFLICT (content): Merge conflict in webserver.conf
Automatic merge failed; fix conflicts and then commit the result.

Step 6: Open webserver.conf, resolve the conflict by keeping the desired port, remove conflict markers, then complete the merge.

# Edit the file to resolve — keep port 443
vim webserver.conf
git add webserver.conf
git commit -m "Merge feature/ssl-config, resolve port conflict to 443"
git branch -d feature/ssl-config

Chapter 2: Creating, Configuring, and Troubleshooting Services

2.1 The systemd Init System

systemd is the system and service manager for modern Linux distributions. It is responsible for bootstrapping the user space and managing services, and it replaces the legacy SysV init system. Key components include:

  • systemd — PID 1, the main init daemon
  • systemctl — The primary command-line tool for interacting with systemd
  • journald — The logging subsystem (journalctl)
  • logind — Login and session management
  • udevd — Device management
  • networkd / resolved — Network and DNS management (optional)

2.2 Unit Files

Everything systemd manages is called a unit. Unit files are plain-text configuration files that describe how systemd should manage a resource.

2.3 Unit File Anatomy

A service unit file consists of three main sections:

[Unit]
Description=My Custom Application Service
Documentation=https://example.com/docs
After=network.target
Requires=network.target
Wants=logging.service

[Service]
Type=simple
ExecStart=/usr/local/bin/myapp --config /etc/myapp/config.yml
ExecStop=/bin/kill -SIGTERM $MAINPID
ExecReload=/bin/kill -SIGHUP $MAINPID
Restart=on-failure
RestartSec=5
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
Environment=ENV=production
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target

Key [Unit] Section Directives

DirectiveDescription
Description=Human-readable description of the unit
After=Units that must be started before this one (ordering only)
Before=Units that must start after this one (ordering only)
Requires=Hard dependency — if the required unit fails, this unit also fails
Wants=Soft dependency — the wanted unit is started but failure is ignored
Conflicts=Units that cannot run simultaneously with this one

Key [Service] Section Directives

DirectiveDescription
Type=simpleDefault. The process started by ExecStart is the main process.
Type=forkingService forks into background. Use PIDFile= to track.
Type=oneshotProcess exits after completing. Used for scripts/setup tasks.
Type=notifyService sends sd_notify() when ready.
Type=dbusService acquires a D-Bus name when ready.
ExecStart=Command to start the service (must be absolute path)
ExecStop=Command to stop the service
ExecReload=Command to reload the service configuration
Restart=When to restart: no, on-success, on-failure, on-abnormal, on-abort, always
RestartSec=Seconds to wait before restarting
User= / Group=Run the service as the specified user/group
WorkingDirectory=Set the working directory for the service
Environment=Set environment variables
EnvironmentFile=Read environment variables from a file

⚑ Exam Alert

You must know how to write a complete systemd unit file from scratch on the exam. Pay particular attention to the Type= directive — choosing the wrong type is one of the most common causes of service failures.

2.9 Troubleshooting Failed Services

When a service fails to start, follow this systematic approach:

  1. Check the status: systemctl status servicename
  2. Read the journal: journalctl -u servicename -n 50 --no-pager
  3. Verify the unit file: systemctl cat servicename
  4. Check ExecStart path exists and is executable: ls -la /path/to/binary
  5. Check file permissions and ownership
  6. Verify dependencies: systemctl list-dependencies servicename
  7. Check SELinux/AppArmor denials (see Chapter 26)
  8. Test the command manually as the service user

RestartSec=5

[Install]
WantedBy=multi-user.target

Chapter 3: Monitoring and Troubleshooting System Performance

3.1 The USE Method

The USE method, developed by Brendan Gregg, provides a systematic approach to performance analysis by examining three metrics for each system resource:

  • Utilization: The percentage of time the resource is busy (e.g., CPU at 90%)
  • Saturation: The degree to which the resource has extra work queued (e.g., run queue length)
  • Errors: The count of error events (e.g., disk I/O errors, network packet drops)

3.2 CPU Monitoring

Understanding Load Averages

$ uptime
 14:32:07 up 12 days,  3:42,  2 users,  load average: 2.50, 1.80, 0.95

$ cat /proc/loadavg
2.50 1.80 0.95 3/285 12847

Load averages represent the average number of processes in a runnable or uninterruptible state over 1, 5, and 15 minutes. For a system with N CPU cores, a load average at or below N indicates the system is not overloaded.

Chapter 4: Application and Service Constraints

4.1 Resource Limits with ulimit

ulimit controls per-process resource limits enforced by the kernel. There are two types:

  • Soft limits: The current enforced limit. Processes can raise soft limits up to the hard limit.
  • Hard limits: The ceiling. Only root can raise hard limits.
# View all current limits
ulimit -a

# View a specific limit
ulimit -n    # Open files (soft)
ulimit -Hn   # Open files (hard)

# Temporarily set a limit (current shell session)
ulimit -n 65536

Persistent Limits: /etc/security/limits.conf

4.3 SELinux Basics

Security-Enhanced Linux (SELinux) is a Mandatory Access Control (MAC) system that adds an additional layer of security beyond traditional Unix permissions.

# Check SELinux status
getenforce        # Returns Enforcing, Permissive, or Disabled
sestatus          # Detailed status

# Temporarily change mode (does not survive reboot)
sudo setenforce 0   # Permissive
sudo setenforce 1   # Enforcing

# Permanent change: edit /etc/selinux/config
# SELINUX=enforcing|permissive|disabled

# View file contexts
ls -Z /var/www/html/

# View process contexts
ps -eZ | grep httpd

# Change file context (temporary)
chcon -t httpd_sys_content_t /var/www/html/index.html

# Restore default context
restorecon -Rv /var/www/html/

# Manage booleans
getsebool -a | grep httpd
setsebool -P httpd_can_network_connect on

# Troubleshoot denials
ausearch -m avc -ts recent
audit2why < /var/log/audit/audit.log

Chapter 6: Working with SSL Certificates

6.1 PKI Fundamentals

Public Key Infrastructure (PKI) is the framework that enables secure communication over networks. Core concepts:

  • Public/Private Key Pair: The public key encrypts data; only the corresponding private key can decrypt it.
  • Certificate Authority (CA): A trusted entity that signs digital certificates, vouching for the identity of the certificate holder.
  • Certificate Signing Request (CSR): A message sent to a CA requesting a signed certificate.
  • Chain of Trust: Root CA → Intermediate CA → Server Certificate. Clients trust the root CA and validate down the chain.

6.2 X.509 Certificate Anatomy

FieldDescription
SubjectThe entity the certificate identifies (CN, O, OU, C, ST, L)
IssuerThe CA that signed the certificate
ValidityNot Before / Not After dates
Public KeyThe subject's public key and algorithm
Serial NumberUnique identifier assigned by the CA
Subject Alternative Name (SAN)Additional hostnames/IPs the cert covers
Signature AlgorithmAlgorithm used by the CA to sign (e.g., SHA256withRSA)

6.3 OpenSSL Comprehensive Guide

Generating Private Keys

Creating Certificate Signing Requests (CSRs)

Format Conversion

# PEM to DER

# DER to PEM

Chapter 10: Access Control Lists (ACLs)

10.1 Why ACLs Are Needed

Standard Unix permissions (owner/group/other) can only assign permissions to one user and one group per file. ACLs allow fine-grained permission assignments to multiple users and groups on a single file or directory.

10.2 Managing ACLs

# View ACLs
getfacl /srv/project/

# Set an ACL for a specific user
setfacl -m u:alice:rwx /srv/project/

# Set an ACL for a specific group
setfacl -m g:developers:rx /srv/project/

# Remove a specific ACL entry
setfacl -x u:alice /srv/project/

# Remove all ACLs
setfacl -b /srv/project/

# Set default ACL (for new files in a directory)
setfacl -dm u:alice:rwx /srv/project/
setfacl -dm g:developers:rx /srv/project/

# Recursive ACL
setfacl -Rm u:alice:rwx /srv/project/

# Backup and restore ACLs
getfacl -R /srv/project/ > acl-backup.txt
setfacl --restore=acl-backup.txt

ACL Mask

The mask defines the maximum permissions that ACL entries (named users, named groups, group-owner) can have. It acts as a ceiling:

# Set the mask
setfacl -m m::rx /srv/project/
# Now even if alice has rwx ACL, effective permissions are limited to rx

11.1 LDAP Fundamentals

LDAP (Lightweight Directory Access Protocol) provides centralized authentication and user/group management. Key concepts:

  • DN (Distinguished Name): Unique identifier, e.g., uid=jdoe,ou=People,dc=example,dc=com

Chapter 23: System Recovery

23.1 Linux Boot Process

Understanding the boot process is essential for recovery:

  1. BIOS/UEFI: Hardware initialization, POST, loads bootloader from boot device
  2. GRUB2: Displays boot menu, loads kernel and initramfs
  3. Kernel: Initializes hardware, mounts initramfs as temporary root
  4. initramfs: Loads necessary drivers, finds real root filesystem
  5. systemd (PID 1): Starts services according to default target

23.3 Root Password Recovery

⚑ Exam Alert

Root password recovery is a high-probability exam task. Practice this procedure until you can do it from memory.

Step-by-step procedure:

  1. Reboot the system and interrupt GRUB by pressing e at the boot menu.
  2. Find the line starting with linux (or linux16) and append rd.break at the end.
  3. Press Ctrl+X to boot with the modified parameters.
  4. You will be dropped into an initramfs emergency shell. The real root filesystem is mounted read-only at /sysroot.
# Remount /sysroot as read-write
mount -o remount,rw /sysroot

# Change root into /sysroot
chroot /sysroot

# Reset the root password
passwd root

# If SELinux is enabled, create autorelabel file
touch /.autorelabel

# Exit chroot and reboot
exit
exit    # or reboot

Containers provide lightweight, isolated environments for running applications. Unlike VMs, containers share the host kernel and use Linux kernel features for isolation:

Chapter 26: SELinux (MAC Enforcement)

26.1 SELinux Architecture

SELinux implements Mandatory Access Control through the Linux Security Module (LSM) framework. The targeted policy (default on RHEL-based systems) confines specific system daemons while leaving user processes unconfined.

26.2 SELinux Contexts

Every file, process, and port has a security context in the format: user:role:type:level

# View file contexts
ls -Z /var/www/html/
# -rw-r--r--. root root unconfined_u:object_r:httpd_sys_content_t:s0 index.html

# View process contexts
ps -eZ | grep httpd
# system_u:system_r:httpd_t:s0  1234 ?  00:00:01 httpd

# View port contexts
sudo semanage port -l | grep http

26.3 Managing File Contexts

# Temporary change (lost on restorecon or relabel)
sudo chcon -t httpd_sys_content_t /srv/web/index.html

# Permanent change (survives restorecon)
sudo semanage fcontext -a -t httpd_sys_content_t "/srv/web(/.*)?"
sudo restorecon -Rv /srv/web/

# List all custom file context rules
sudo semanage fcontext -l -C

# List booleans with descriptions

27.3 Extending Logical Volumes

⚑ Exam Alert

Extending an LV is a two-step process: extend the LV, then extend the filesystem. Using lvextend -r does both in one step. This is the most commonly tested LVM task.

# Extend LV by 5GB and resize filesystem in one step
sudo lvextend -r -L +5G /dev/datavg/datalv

# Or do it manually:
# Step 1: Extend the LV
sudo lvextend -L +5G /dev/datavg/datalv

# Step 2: Resize the filesystem
sudo resize2fs /dev/datavg/datalv     # ext4
sudo xfs_growfs /mountpoint            # XFS (use mount point, not device)

⚠ Warning

XFS filesystems can only be grown, never shrunk. ext4 filesystems can be both grown and shrunk, but shrinking requires unmounting first and running fsck.

27.4 Reducing Logical Volumes (ext4 only)

# Shrinking requires: unmount → fsck → resize2fs → lvreduce
sudo umount /data
sudo e2fsck -f /dev/datavg/datalv
sudo resize2fs /dev/datavg/datalv 5G
sudo lvreduce -L 5G /dev/datavg/datalv
sudo mount /dev/datavg/datalv /data

# Revert to snapshot (merge)
sudo umount /data
sudo lvconvert --merge /dev/datavg/datasnap

Appendix E: Troubleshooting Decision Trees

Service Won't Start

Service won't start
├── Run: systemctl status servicename
│   ├── Status: 203/EXEC
│   │   └── ExecStart binary not found or not executable
│   │       ├── Check path: which binary_name
│   │       ├── Check permissions: ls -la /path/to/binary
│   │       └── Fix: correct path or chmod +x
│   ├── Status: 217/USER
│   │   └── User specified in unit file does not exist
│   │       └── Fix: create user or change User= directive
│   ├── Status: exit-code or signal
│   │   └── Application error
│   │       ├── Check: journalctl -u servicename
│   │       ├── Test manually: run ExecStart command as service user
│   │       └── Check config files for the application
│   └── Dependency failure
│       ├── Check: systemctl list-dependencies servicename
│       └── Ensure After= and Requires= units are running
├── After fixing: systemctl daemon-reload
└── Restart: systemctl restart servicename

Network Connectivity Issues

Cannot reach remote host
├── Check interface: ip link show
│   └── Interface DOWN? → ip link set dev eth0 up
├── Check IP address: ip addr show
│   └── No IP? → nmcli connection up connection_name
├── Check routing: ip route show
│   └── No default gateway? → ip route add default via GATEWAY
├── Ping gateway: ping GATEWAY_IP
│   └── Fails? → Layer 2 issue (cable, switch, VLAN)
├── Ping external IP: ping 8.8.8.8
│   └── Fails? → Routing or firewall issue
│       └── Check: nft list ruleset / firewall-cmd --list-all
├── Ping by hostname: ping example.com
│   └── Fails? → DNS issue
│       ├── Check: cat /etc/resolv.conf
│       ├── Test: dig example.com @8.8.8.8
│       └── Fix DNS config via nmcli
└── Check specific port: ss -tlnp | grep PORT
    └── Service not listening? → Check service status

Disk Space Issues

Disk full (df shows 100%)
├── Identify largest directories: du -h --max-depth=1 / | sort -rh
├── Check for deleted-but-open files: lsof +L1 | grep deleted
│   └── Found? → Restart the process holding the file
├── Clean package caches: apt clean / dnf clean all
├── Clean old journals: journalctl --vacuum-size=100M
├── Check for old logs: find /var/log -name "*.gz" -mtime +90
├── Check /tmp: find /tmp -type f -atime +7
└── If inode exhaustion (df -i shows 100%):
    ├── Find dirs with most files: find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn
    └── Remove unnecessary small files

Boot Failure

System won't boot
├── GRUB menu appears?
│   ├── Yes → Edit kernel line (press 'e')
│   │   ├── Add systemd.unit=rescue.target for rescue mode
│   │   ├── Add rd.break for initramfs shell (password reset)
│   │   └── Add init=/bin/bash for emergency shell
│   └── No → GRUB is corrupted
│       ├── Boot from live media
│       ├── Mount root filesystem
│       ├── chroot into it
│       └── Reinstall GRUB: grub2-install /dev/sda
├── Kernel panic?
│   └── Boot previous kernel from GRUB menu
├── Filesystem errors?
│   ├── Boot to emergency mode
│   ├── fsck the affected filesystem
│   └── Reboot
└── Service failure preventing login?
    ├── Boot to rescue.target
    ├── Fix the service (disable or correct it)
    └── Reboot to normal target

Permission Denied

Permission denied error
├── Check standard permissions: ls -la file
│   └── Wrong permissions? → chmod / chown
├── Check ACLs: getfacl file
│   └── Restrictive ACL mask? → setfacl -m m::rwx file
├── Check SELinux (RHEL-based):
│   ├── getenforce → Enforcing?
│   ├── ls -Z file → Wrong context?
│   │   └── restorecon -Rv /path or semanage fcontext
│   ├── Check booleans: getsebool -a | grep service
│   └── Check denials: ausearch -m avc -ts recent | audit2why
├── Check AppArmor (Ubuntu):
│   └── aa-status → Profile in enforce mode?
├── Check mount options: mount | grep noexec
│   └── noexec on mount? → Remount without noexec
└── Check capabilities: getcap binary
